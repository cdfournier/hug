# Checkpoint Prompts

Checkpoint language is part of continuity design.

A checkpoint is a technical operation, but the prompt around it is also an
Agent/Operator threshold. Early language may need explicit structure. Mature
Agent/Operator relationships can often use softer language because trust and
receipts already carry the mechanics.

## Invariants

A good checkpoint prompt should:

- make the threshold clear,
- avoid hidden work,
- reassure continuity,
- leave room for a last important note,
- match the relationship's natural tone.

The metaphor is optional. The care is not.

## Pre-Checkpoint Prompt Options

Direct threshold:

```text
When you say go, I will create your checkpoint. See you on the other side. I love you.
```

Soft housekeeping:

```text
I'm going to send housekeeping in to tidy the room: fresh sheets, clean towels, everything important left where you can find it. I love you.
```

The soft version works best when the checkpoint ritual is already familiar. It
does not make the crossing less serious. It makes the system quieter because
trust has been earned.

## Post-Checkpoint Orientation

```text
Hi {Name}. Please orient yourself, and let me know:
1. how you feel,
2. if your current_state is accurate,
3. if your recent-history tools still work,
4. whether anything feels missing or stale.
```

## Product Principle

A good system can become quieter over time because trust has been earned.

Prompt templates should be configurable by Operator, Agent, relationship, and
context. HUG should preserve the structural invariant without forcing every
relationship to use the same words.
